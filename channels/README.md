(задания по нарастающей сложности — от первого `MethodChannel` до `Texture` и `Pigeon`).

## Часть I. Теория

### 1. Как Flutter рисует виджеты и причём тут каналы

Flutter не использует нативные виджеты ОС. Он рисует пиксели сам, своим движком (Skia/Impeller).
Пайплайн: `Widget` (конфиг, immutable) → `Element` (дерево, lifecycle) → `RenderObject`
(layout + paint) → движок отдаёт кадр на GPU.

Ключевая мысль: рендер виджетов **самодостаточен и не использует каналы**. Канал появляется
не для UI, а когда задача выходит за пределы экрана — в платформенные API (GPS, push, карта,
биометрия). Это две независимые оси.

### 2. Простейшие примеры: батарея и геолокатор

- **Батарея → MethodChannel.** Dart спрашивает — натив отвечает один раз. Инициатор — Dart.
- **Геолокатор → EventChannel.** Натив сам шлёт поток координат. Инициатор — натив.

Правило большого пальца: разовый ответ → `MethodChannel`; поток событий во времени → `EventChannel`.
90% популярных плагинов (`geolocator`, `battery_plus`, `firebase_messaging`) внутри — ровно эти связки.

### 3. Сколько способов связать Dart и натив

Две независимые оси.

**A. Каналы — 3 типа:**

| Канал | Инициатор | Когда |
|---|---|---|
| `MethodChannel` | Dart | запрос/ответ, императивные команды |
| `EventChannel` | натив | поток событий во времени |
| `BasicMessageChannel` | обе стороны | симметричный обмен без семантики «метод» |

**B. Нативный UI — 2 способа:**

- **Platform View** (`AndroidView` / `UiKitView`) — встраивает нативный `View` в дерево Flutter.
- **Texture** (`TextureView` / `SurfaceTexture`) — натив рисует в текстуру, Flutter компонует её как слой.

Плюс системные механизмы: **FFI** (прямой вызов C/C++, без нативного слоя и без перехода на
platform thread) и **Pigeon** (кодген поверх каналов — не отдельный тип, а типобезопасная обёртка над `MethodChannel`).

### 4. Кто инициатор — это и есть выбор канала

Выбор делается **не по типу данных, а по тому, кто владеет инициативой и временем жизни подписки.**
Натив инициирует → `EventChannel`. Dart инициирует → `MethodChannel`.

Антипаттерн уровня код-ревью: эмуляция потока через периодический `invokeMethod` (polling) —
лишние пробуждения, расход батареи, latency. Если источник событий — натив, это всегда `EventChannel`.

### 5. Нативный UI на Android: почему view застревал поверх

Это реальная боль всего Flutter-сообщества (десятки GitHub issues), а не выдуманный кейс.

**Иерархия (важно для точности):** `AndroidView` — это Dart-виджет («что показываем»).
А **режим композиции** («как движок встраивает нативный view») — это одно из:

- **Virtual Display** — нативный view рендерится в невидимый виртуальный экран, показывается как
  текстура. Проблемы с тачем, доступностью, клавиатурой.
- **Hybrid Composition (HC)** — реальный нативный `View` встраивается в иерархию Android поверх
  `FlutterView`. Тач и доступность правильные, но именно эта «двухвладельческая» модель порождает баги.
- **TLHC (Texture Layer Hybrid Composition)** — в современных версиях Flutter дефолтный компромисс:
  нативный view в текстуре, но с корректным вводом.

Режим выбирается **способом инициализации** (`initSurfaceAndroidView` / `initExpensiveAndroidView` /
старый `AndroidView`), а **не конфигом Gradle или манифеста.**

`Texture` — это **отдельная ветка**, не режим `AndroidView`. Там нативный view вообще не встраивается:
натив пишет в `SurfaceTexture`, Flutter показывает результат как обычный слой.

**Почему view застревал именно на Android (Hybrid Composition):** реальный нативный `View` живёт в
иерархии Android, владелец — Android, не Flutter. При повторном заходе `Element` не уничтожался, и
осиротевший нативный view оставался в иерархии с z-order выше `FlutterView` — карта висела поверх UI.

**Почему на iOS такого нет:** там Platform View всегда под контролем Flutter-композитора (z-order
пересчитывается на каждый кадр). Двухвладельческой модели, как в HC, на iOS нет.

**Почему спасал `Navigator.pop`:** `pop` уничтожает `Route` целиком — а с ним `Element`
платформенного view, что гарантированно запускает teardown нативного view. Обычный `setState` мог
переиспользовать тот же `Element`, и view не убирался. То есть `pop` маскировал симптом; правильный
фикс — симметричный `dispose()` и/или переход на `Texture`-подход.

**Боевое решение:** переход на официальный пакет 2GIS, работающий через `Texture`, где осиротевший
слой невозможен по построению.

### 6. Сколько каналов нужно для нативного view

«View-каналов» как типа не существует. Регистрируется **фабрика view**, а свои каналы добавляются вручную.

| Что | Android | iOS |
|---|---|---|
| Регистрация view | `PlatformViewFactory` (фабрика, не канал) | `FlutterPlatformViewFactory` (фабрика, не канал) |
| Системный канал движка | `flutter/platform_views` (внутренний) | `flutter/platform_views` (внутренний) |
| Твой канал на экземпляр | 1 × `MethodChannel` (`tez/map2gis_$id`) | 1 × `MethodChannel` (`tez/map2gis_$id`) |
| Поток событий (опц.) | 1 × `EventChannel` при необходимости | 1 × `EventChannel` при необходимости |

Число каналов не зависит от платформы — оно зависит от числа карт на экране. Две карты → два инстанс-канала.

### 7. StandardMessageCodec — за что отвечает

Кодек сериализует Dart-значения в байты для передачи через канал и обратно.

Из коробки: `null`, `bool`, `int`, `double`, `String`, `Uint8List`/`Int32List`/`Float64List`,
а также `List` и `Map` из этих типов. Свой класс напрямую **не передаётся** — его нужно разложить
в `Map` либо написать кастомный кодек: наследуешь `StandardMessageCodec`, переопределяешь
`writeValue` / `readValueOfType`, выделяешь свой type-tag (≥ 128) под доменный объект и передаёшь
кодек в конструктор канала.

### 8. Трудности

- **Lifecycle** — движки, сцены, Activity, view пересоздаются независимо. Половина «плавающих» багов отсюда.
- **Гонки инициализации** — натив и Dart стартуют асинхронно; «работает у меня» = повезло с таймингом.
- **Потоки** — хендлер по умолчанию на UI-потоке платформы; jank не виден Dart DevTools.
- **Тестируемость** — канал не замокать в виджет-тесте напрямую; интеграционные сценарии — на устройстве.

### 9. Pigeon — типобезопасный контракт

Описываешь контракт на Dart (`@HostApi` / `@FlutterApi`), кодген строит обёртки для Dart, Kotlin и
Swift. Уходит целый класс ошибок: строковые имена, рассинхрон сигнатур, ручная сериализация. Для
нового нативного кода — разумный дефолт. Аргумент для senior: это снижение стоимости владения (TCO),
а не «синтаксический сахар».

### 10. Background platform channels

По умолчанию нативный хендлер исполняется на главном (UI) потоке платформы. Тяжёлая работа в
хендлере фризит натив, и Dart DevTools этого не показывает. С Flutter 3.7+ можно создать
`TaskQueue` (`makeBackgroundTaskQueue()`) и передать его в `setMethodCallHandler` — хендлер уйдёт на
отдельный поток платформы, UI-поток останется свободным.

### 11. CarPlay — отдельный мир

CarPlay не показывает обычный Flutter-UI: у него свои шаблоны (`CPMapTemplate`, `CPListTemplate`),
взаимодействие — по правилам Apple для вождения. Верстать Flutter-виджеты для CarPlay нельзя.

Но бизнес-логику на Dart **не пишут с нуля**: поднимается отдельный `FlutterEngine` для CarPlay-сцены,
состояние заказа течёт по `EventChannel`, а нативный CarPlay-код — это тонкий слой представления над
общим Dart-ядром. Грабли мульти-движка: канал телефонного движка в CarPlay-движке не существует
(«на телефоне работает, в машине молчит») — каналы регистрируются на каждом движке отдельно.

### 12. Любая нативка через каналы — это плагин?

Нет. **Канал — это механизм. Плагин — это упаковка.**

- **App-level интеграция** — канал + Kotlin/Swift прямо в модуле приложения (`MainActivity` /
  `AppDelegate`). Не плагин, приватная нативка проекта.
- **Plugin** — отдельный package с `android/`/`ios/`, регистрацией `FlutterPlugin` и объявлением в
  `pubspec`. Переиспользуемый, публикуемый. Может быть и на FFI без каналов.

Критерий: вынес нативную часть в отдельный package с регистрацией под платформы → плагин.

### 13. Путь от Flutter к нативке по чистой архитектуре

```
Presentation  (BLoC / UI — доменный язык)
      ↓
Domain        (UseCase + абстрактный Repository)
      ↓
Data          (RepositoryImpl + Service-интерфейс)
      ↓
Platform      (Channel → нативный SDK)
```

Канал живёт **только в Platform-слое, за интерфейсом Service.** Domain и Presentation про каналы не
знают. Примеры пересечения границы:

- Карта 2GIS: `MapRepository → MapService → Texture + MethodChannel`
- `firebase_messaging`: `PushRepository → плагин → MethodChannel + нативный SDK`
- Геолокатор: `LocationRepository → LocationService → EventChannel`

### 14. У натива свой lifecycle

- **Android:** `Activity` (`onCreate → onResume → onPause → onDestroy`), пересоздаётся при повороте,
  смене permission, нехватке памяти; `FlutterEngine` может пережить `Activity`.
- **iOS:** `AppDelegate`/`SceneDelegate` (foreground/background), CarPlay-сцена (`connect → disconnect`),
  `ViewController` (`viewDidLoad → viewDidDisappear`).

Где больно: Dart-виджет и нативный объект живут по **разным** циклам. `dispose` на стороне Dart ≠
конец нативного объекта. Подписки, view, SDK освобождай явно по нативным событиям.

### 15. Ошибки натива — часть контракта

У `MethodChannel` ровно три исхода: `success`, `error` (→ `PlatformException`), `notImplemented`
(→ `MissingPluginException`). Все три обрабатываются на стороне Dart. `PlatformException` — не баг, а
штатный исход (устройство без железа, отказ в permission, старая версия натива).

### 16. Два принципа на вынос

- **Канал — в data-слой, за интерфейс.** Домен не протекает в натив. Платформа подменяема, код тестируем.
- **Канал = контракт между процессами.** Проектируй как API, версионируй, держи имена в одном месте.

## Часть II. Практика (воркшоп)

Задания идут по нарастающей. Каждое снабжено целью, шагами и критериями приёмки (DoD).
Делать можно на чистом `flutter create` проекте; где нужно — добавляй нативный код в `MainActivity`
(Kotlin) и `AppDelegate`/плагин (Swift).

> Совет: оборачивай каждый канал в сервис-класс (`BatteryService`, `LocationService`, `MapService`).
> Прямой вызов `MethodChannel` из виджета — антипаттерн, который ломает тестируемость.

---

### Уровень 1. MethodChannel

#### Практика 1.1 — Уровень заряда батареи

**Цель:** первый запрос/ответ через канал.

1. Создай `MethodChannel('workshop/battery')`.
2. На Android реализуй метод `getLevel`, верни уровень из `BatteryManager`.
3. На iOS реализуй `getLevel` через `UIDevice.current.batteryLevel`.
4. Оберни вызов в `BatteryService.level()` с обработкой `PlatformException` (возврат `-1`).

**DoD:** на обеих платформах кнопка показывает реальный заряд; при выдернутом методе на одной
платформе приложение не падает, а показывает `-1`.

#### Практика 1.2 — Три исхода

**Цель:** прочувствовать контракт ошибок.

1. Добавь метод `mayFail`, который на нативе с вероятностью 50% возвращает `result.error("E_RANDOM", ...)`.
2. Вызови несуществующий метод `ghost` и поймай `MissingPluginException`.
3. В Dart обработай все три ветки: `success`, `on PlatformException`, `on MissingPluginException`.

**DoD:** UI корректно различает три состояния и выводит понятное сообщение для каждого.

#### Практика 1.3 — Передача структур

**Цель:** понять границы `StandardMessageCodec`.

1. Передай в натив `Map{'a': 1, 'b': [1,2,3], 'c': 'x'}` и верни его обратно изменённым.
2. Попробуй передать свой класс `User(name, age)` напрямую — убедись, что не работает.
3. Разложи `User` в `Map` и передай корректно.

**DoD:** структура ходит туда-обратно; ты можешь объяснить, почему класс напрямую не прошёл.

### Уровень 2. EventChannel

#### Практика 2.1 — Таймер-тик

**Цель:** первый поток событий от натива.

1. Создай `EventChannel('workshop/ticker')`.
2. На нативе раз в секунду шли в `EventSink` инкрементируемое число.
3. В Dart подпишись и выводи на экран; в `dispose` отпишись.

**DoD:** числа идут раз в секунду; при уходе с экрана нативный таймер останавливается
(проверь логом в `onCancel`).

#### Практика 2.2 — Сенсор / геолокация

**Цель:** реальный поток данных.

1. Сделай `LocationService.stream()` поверх `EventChannel`, маппи в модель `LatLng`.
2. На нативе подпишись на обновления локации и шли координаты.
3. Корректно обработай отказ в permission: пробрось ошибку в `EventSink` (`sink.error(...)`).

**DoD:** координаты текут; при отказе в permission поток получает ошибку, а не молча умирает.

#### Практика 2.3 — Гонка cold start

**Цель:** воспроизвести и починить классический баг.

1. На нативе начни слать события **сразу** в `onListen`, до того как Dart точно готов.
2. Убедись, что первое событие иногда теряется.
3. Почини: буферизуй последнее значение и отдавай его новому слушателю (или введи явный сигнал готовности из Dart).

**DoD:** первое событие больше не теряется; ты можешь объяснить, почему гонка возникала.

### Уровень 3. BasicMessageChannel

#### Практика 3.1 — Эхо

**Цель:** симметричный обмен.

1. Создай `BasicMessageChannel<String>('workshop/echo', StringCodec())`.
2. Dart шлёт строку — натив отвечает той же строкой в верхнем регистре.
3. Натив инициативно шлёт «ping» — Dart отвечает «pong».

**DoD:** обе стороны умеют и слать, и отвечать; в логе видно оба направления.

#### Практика 3.2 — Sync темы

**Цель:** практический кейс state-sync.

1. Через `BasicMessageChannel` синхронизируй тему (`light`/`dark`) между Dart и нативным экраном.
2. Смена темы в Dart применяется на нативе и наоборот.

**DoD:** изменение с любой стороны мгновенно отражается на другой.

### Уровень 4. Platform Views

#### Практика 4.1 — Нативная кнопка в дереве Flutter

**Цель:** первый Platform View.

1. На Android зарегистрируй `PlatformViewFactory`, возвращающую нативную `Button`.
2. На iOS зарегистрируй `FlutterPlatformViewFactory` с `UIButton`.
3. В Dart покажи через `AndroidView` / `UiKitView`.

**DoD:** нативная кнопка видна и кликается на обеих платформах.

#### Практика 4.2 — Канал на экземпляр

**Цель:** связь Dart ↔ конкретный нативный view.

1. В `onPlatformViewCreated(id)` создай `MethodChannel('workshop/btn_$id')`.
2. Команда из Dart `setTitle('Hi')` меняет текст нативной кнопки.
3. Клик по нативной кнопке шлёт событие обратно в Dart.

**DoD:** два инстанса view на одном экране управляются независимо через свои каналы.

#### Практика 4.3 — Воспроизведи баг застревания (Android)

**Цель:** понять проблему Hybrid Composition на практике.

1. Используй Platform View в режиме Hybrid Composition.
2. Сделай навигацию: экран со view → push другого экрана → возврат, и повтори несколько раз.
3. Добейся ситуации, когда нативный view «залипает» поверх (или артефакты композиции).
4. Сравни поведение с `Navigator.pop` против простого скрытия виджета.

**DoD:** ты можешь продемонстрировать разницу и объяснить роль уничтожения `Element`.

#### Практика 4.4 — Симметричный dispose

**Цель:** правильный фикс lifecycle.

1. Реализуй `dispose()` у нативного `PlatformView`: останови ресурсы, обнули хендлер канала,
   сними view из иерархии.
2. Убедись, что view убирается при любом уходе с экрана, а не только при `pop`.

**DoD:** баг из 4.3 больше не воспроизводится без костыля с `pop`.

**Цель:** альтернатива Platform View.

#### Практика 5.2 — Сравнительный анализ

### Уровень 6. Pigeon

#### Практика 6.1 — Контракт вместо строк

**Цель:** типобезопасный канал.

1. Опиши `@HostApi() abstract class CalcApi { int add(int a, int b); }`.
2. Сгенерируй обёртки (`dart run pigeon ...`).
3. Реализуй нативную часть (Kotlin interface / Swift protocol) и вызови из Dart.

**DoD:** вызов типобезопасный; намеренная опечатка в имени метода теперь не компилируется.

#### Практика 6.2 — Двусторонний (`@FlutterApi`)

**Цель:** обратные вызовы из натива в Dart.

1. Добавь `@FlutterApi() abstract class CalcEvents { void onResult(int value); }`.
2. Натив инициативно дёргает `onResult`, Dart его реализует.

**DoD:** видно оба направления, всё типобезопасно, без строковых имён.

#### Практика 8.1 — Канал за интерфейсом

## Чек-лист «забрать с собой»

- [ ] Канал = контракт между процессами. Проектируй как API, версионируй.
- [ ] Pigeon для нового кода — снижение TCO, не «сахар».
- [ ] Канал — в data-слой, за интерфейс. Домен не протекает в натив.
- [ ] Нативный хендлер — диспетчер, не воркер. Знай про background channels.
- [ ] Lifecycle симметричен: `create ↔ dispose`. Закрывай подписки и снимай view.
- [ ] Ошибки натива (`PlatformException`, `notImplemented`) — часть контракта.
- [ ] `AndroidView` ≠ режим композиции: VD / HC / TLHC — это «как», а не «что». `Texture` — отдельная ветка.
- [ ] Инициатор определяет канал: натив → EventChannel, Dart → MethodChannel.
